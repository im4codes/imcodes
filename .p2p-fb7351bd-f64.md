# P2P Discussion: fb7351bd-f64

## User Request

Gemini 老是不停working idle状态切换 修复一下

## brain — Initial Analysis

### 问题根因

Gemini watcher 有两个状态源在 500ms 轮询中互相打架：

1. **JSON conversation status**（`inferConversationStatus`）— 读 Gemini session JSON
2. **Terminal detection**（`terminalThinkingCheck`）— 捕获 tmux pane + `detectStatus`

**关键 bug 在 `pollTick` 第 228-232 行**：

```typescript
if (conv.lastUpdated === state.lastUpdated && conv.messages.length === state.seenCount) {
    if (conversationStatus === 'idle') {
      state.emittedRunning = false;  // ← 直接重置，无 debounce！
      return;
    }
    await terminalThinkingCheck(sessionName, state);
    return;
}
```

当 JSON 未变化且 conversation status 是 idle 时，`state.emittedRunning` 立即被设为 false，然后下一个 poll tick terminal check 可能检测到 activity 又设为 true → emit running。循环往复造成 working/idle 闪烁。

### 修复方案

1. **统一 debounce**：JSON idle 也走 debounce 路径，不直接重置 `emittedRunning`
2. **idle 状态锁定**：一旦 emit idle，短期内（如 2s）不响应 terminal activity，避免 terminal 瞬态误判触发 running
3. **减少 poll 频率**：500ms 太频繁，对于状态检测 1-2s 就够了

### 受影响代码

- `src/daemon/gemini-watcher.ts:228-236` — JSON unchanged + idle 路径
- `src/daemon/gemini-watcher.ts:143-156` — terminal idle debounce
- `src/daemon/gemini-watcher.ts:256-282` — JSON changed + status emit

## 2p3z423w — Audit (hop 1/2)

### 审计结论

brain 已经指出了核心逻辑 bug，我补充成更完整的“风险/后果/修复优先级”视角：这不是单纯的 UI 闪烁问题，而是 **状态机有两个来源且缺少统一仲裁**，会带来逻辑一致性风险。

### HIGH 1: JSON idle 分支直接清掉 running 标志，绕过所有 debounce

**位置**：`src/daemon/gemini-watcher.ts:228-236`

当前逻辑：
```ts
if (conv.lastUpdated === state.lastUpdated && conv.messages.length === state.seenCount) {
  if (conversationStatus === 'idle') {
    state.emittedRunning = false;
    return;
  }
  await terminalThinkingCheck(sessionName, state);
  return;
}
```

问题：
- 这里直接把 `state.emittedRunning = false`
- 没有经过 `idleDebounceTimer`
- 也没有真正 `emitSessionState(sessionName, 'idle')`
- 只是把内部状态改掉了

后果：
- 下一次 `terminalThinkingCheck()` 只要看到一点 activity，就会因为 `!state.emittedRunning` 再次发 `running`
- 于是内部状态在“idle / running”之间来回摆，外部表现就是 working / idle 闪烁
- 更严重的是：**内部状态和实际已发出的 timeline state 可能不一致**

这是这次问题的主根因。

### HIGH 2: watcher 同时依赖两个状态源，但没有 source-of-truth 规则

**位置**：
- `src/daemon/gemini-watcher.ts:133-171`（terminalThinkingCheck）
- `src/daemon/gemini-watcher.ts:225-281`（JSON conversationStatus 分支）

现状：
- JSON conversation status 会驱动 idle/running
- terminal detection 也会驱动 running/idle
- 两者都在改 `state.emittedRunning`
- 但没有统一规则说明：
  - 谁优先
  - 谁只能补充 thinking
  - 谁能真正 emit idle

风险：
- 这是典型的双写状态源问题
- 只要一个源是 debounce、另一个源是即时写，就必然出现竞争
- 后续再调 poll interval 或 idle pattern，很容易引入新抖动

建议：
- 必须定义单一规则，例如：
  - **JSON 是 session.state 的 authoritative source**
  - terminal 只补 `assistant.thinking`
  - terminal 不能在 JSON 已判 idle 后立即重新拉起 running
- 或至少引入“最近一次 idle 来源”和“idle lock”机制

### MEDIUM 3: terminal idle 分支有 debounce，但 JSON idle 分支没有，行为不对称

**位置**：
- `src/daemon/gemini-watcher.ts:143-156`
- 对比 `src/daemon/gemini-watcher.ts:265-281`

terminal 分支：
- idle 会走 3s debounce

JSON changed + idle 分支：
- `isUpdate && touchedGeminiMessage` 时 debounce 1.5s
- 否则立即 idle

JSON unchanged + idle 分支：
- 直接把 `emittedRunning = false`
- 连 emit 都没有

这三条路径的行为完全不一致。

风险：
- 同一个“系统进入 idle”事件，根据数据来源不同，走三套不同状态机
- 维护者很难推断某次 flicker 是哪条路径触发的

建议：
- idle 发射逻辑统一封装成一个函数
- 所有路径都走同一个状态过渡入口
- 不允许直接手改 `emittedRunning`

### MEDIUM 4: poll interval 500ms 加剧竞争窗口

**位置**：`src/daemon/gemini-watcher.ts:307-322, 340-371`

虽然具体常量不在这段，但现有逻辑明显是高频轮询驱动。500ms 对双源状态机来说过于激进，后果是：
- terminal 一点瞬态变化就会被捕获
- JSON 未变化的空档会非常频繁触发 `228-236` 这个有 bug 的分支

风险：
- 轮询越快，状态抖动越明显
- 也会增加 capturePane / readConversation 的负载

建议：
- 在修根因前，不建议单靠调大 interval 解决
- 但修完后，1s~2s polling 对状态检测更稳

### MEDIUM 5: `state.emittedRunning` 同时承担“是否已经 emit”与“当前逻辑状态”两种语义

**位置**：贯穿 `src/daemon/gemini-watcher.ts`，典型点：
- `146, 149-152`
- `163-166`
- `228-231`
- `258-260`
- `273, 279`
- `300-302`

问题：
- 这个字段名字叫 `emittedRunning`
- 含义应该是“是否已经向外发过 running”
- 但代码里实际上把它当作“当前内部状态是不是 running”来用

风险：
- 语义混杂导致很多看似合理的写法其实在破坏状态机
- 比如 `230` 行只把它设成 false，就等价于对外逻辑进入 idle，但实际上并没有发 idle 事件

建议：
- 最好拆成两个字段：
  - `currentState: 'running' | 'idle' | null`
  - `terminalThinkingActive: boolean`
- 或者统一走 `emitSessionStateIfChanged()`，不要手动维护布尔位

### LOW 6: startWatching 初始 idle 恢复路径也绕过了一部分“稳定确认”逻辑

**位置**：`src/daemon/gemini-watcher.ts:292-304`

这里在启动恢复时：
- 读到 `lastConversationStatus === 'idle'`
- 直接 `emitSessionState(sessionName, 'idle')`

这本身不一定错，但结合双源竞争，会带来另一个边界：
- watcher 刚恢复 → 先发 idle
- terminal poll 很快检测到 activity → 再发 running
- 如果 JSON 没马上变化，又可能走 bug 分支清掉 running 标记

结论：
- 这不是主 bug，但说明状态源统一问题在恢复路径也存在

### 推荐修复优先级

#### 必须先修
1. 去掉 `228-231` 的直接 `state.emittedRunning = false`
2. 统一 idle 状态过渡入口，所有 idle 路径都走同一套 debounce / emit 逻辑
3. 明确 JSON 和 terminal 的角色分工，避免双源都能随意改 session.state

#### 强烈建议一起修
4. 给 idle 后短时间增加 lock / cool-down，防止 terminal 瞬态立即拉回 running
5. 降低 poll 频率到更稳妥的区间（例如 1s 或 1.5s）
6. 补测试：
   - JSON unchanged + idle 不应立即清 running 标志
   - 已 emit idle 后，短期 terminal activity 不应立即闪回 running
   - JSON 和 terminal 同时存在时，以哪一方为准

### 最终判断

这不是简单的“Gemini pattern 不准”，而是 **状态机设计问题**：
- 双状态源
- 不一致的 idle 过渡路径
- 一个布尔字段承载了过多语义

如果只做局部修补，很容易再回归。比较稳的处理方式是：
- 先把 session.state 的 authoritative source 定清楚
- 再让 terminal 只做补充信号
- 最后统一 idle/running 的 emit 入口

## 6i2q4d6j — Audit (hop 2/2)

### Technical Analysis: State Machine Fragmentation

The previous audit correctly identified the core issue as a conflict between two unsynchronized state sources. However, the current code in `src/daemon/gemini-watcher.ts` still contains architectural inconsistencies that will likely lead to future regressions.

#### 1. CRITICAL: Incomplete Transition to Unified State Management
While a `transitionState` helper (lines 160-169) was introduced to handle state changes with an "idle lock," it is not used consistently throughout the file.
- **Vulnerability**: The `pollTick` function (lines 257-282) still manually modifies `state.emittedRunning` and calls `emitSessionState` directly.
- **Risk**: Transitions triggered by JSON updates bypass the `IDLE_LOCK_MS` safety mechanism. If a JSON update emits `idle` and a terminal check immediately follows, the terminal check might override it with `running` because the lock was never set by the JSON path.

#### 2. HIGH: Dual State tracking (Desynchronization)
The code now maintains two distinct state indicators: `state.currentState` (introduced for `transitionState`) and the original `state.emittedRunning` (retained in `pollTick`).
- **Logic Error**: These two variables can get out of sync. `transitionState` updates `currentState`, but `pollTick` relies on `emittedRunning`. 
- **Impact**: A session could be logically `idle` according to `currentState` but still marked as `running` by `emittedRunning`, leading to skipped checks or incorrect event emissions.

#### 3. MEDIUM: Lack of Authoritative Source Hierarchy
The watcher treats JSON status and terminal status as peers, but they serve different purposes.
- **Recommendation**: JSON should be the **authoritative source** for `session.state`. Terminal detection should be a **supplementary source** that can only upgrade the state to `running` (thinking) but should rarely be allowed to declare the session `idle` if the JSON suggests otherwise, or vice-versa.
- **Improvement**: Implement a priority queue where JSON signals always override terminal signals within a stability window.

#### 4. LOW: Magic Numbers and Polling Overhead
- **Finding**: `POLL_INTERVAL_MS` is set to 1500ms, and `IDLE_LOCK_MS` to 2000ms.
- **Observation**: These numbers are tightly coupled. If the lock window is too short relative to the polling interval, the flicker will return. The current values are safe but brittle.

### Summary
The fix is **partially effective** but architecturally **fragile**. To ensure long-term stability, the `emittedRunning` flag should be entirely removed in favor of `currentState`, and **all** state transitions (including those from JSON parsing) must pass through the `transitionState` arbitrator.
