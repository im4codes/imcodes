### 安全审计发现

#### HIGH: temp file 内容注入风险 (timeline-emitter.ts)

```typescript
const TEMP_FILE_RE = /^Read and execute all instructions in @(.+\.imcodes-prompt-[0-9a-f]+\.md)$/;
```

Timeline emitter 检测到匹配 pattern 的 user.message 时，会用 readFileSync 读取文件内容并替换到 payload。

**风险**：如果攻击者能控制 user.message 的 text 字段（例如通过 chat platform webhook），可以构造一个匹配 pattern 的消息来读取任意 .imcodes-prompt-*.md 文件。

**缓解**：pattern 已经限制了文件名格式（.imcodes-prompt-[0-9a-f]+.md），且 readFileSync 失败时不会泄露错误信息。风险较低但存在。

**建议**：可以额外验证文件路径在 /tmp/ 或 projectDir 下，防止路径穿越。

#### MEDIUM: temp file 竞态窗口 (tmux.ts)

```typescript
await fsp.writeFile(filePath, keys, 'utf-8');
// ... send instruction ...
setTimeout(() => fsp.unlink(filePath).catch(() => {}), 120_000);
```

temp file 在写入后 120 秒才删除。在这个窗口内，同一机器上的其他用户（如果有）可以读取 /tmp/.imcodes-prompt-*.md 文件，其中可能包含敏感的 P2P 讨论内容。

**缓解**：文件名包含 hash，不可预测。保持 /tmp 是全局可读的。

**建议**：使用 0o600 权限创建文件：await fsp.writeFile(filePath, keys, { encoding: 'utf-8', mode: 0o600 })。

#### MEDIUM: dispatchHop retry 可能发送重复 prompt

```typescript
for (let attempt = 0; attempt <= MAX_RETRIES; attempt++) {
  await sendKeysDelayedEnter(session, prompt, sendOpts);
  // ... wait for completion ...
  // idle without growth → retry
}
```

如果 agent 实际上收到了第一次 prompt 但文件写入延迟（例如磁盘 I/O），retry 会发送第二次 prompt。agent 可能执行两次同样的任务。

**缓解**：MAX_RETRIES=1，最多重复一次。Grace period 30s 降低了误判概率。

**建议**：可接受的风险。P2P 讨论重复写入不会造成数据损坏，只是浪费 token。

#### LOW: Gemini idle pattern 过于宽泛

```typescript
/Type your message or @/m
```

这个 pattern 只匹配 Type your message or @，但如果 agent 输出中恰好包含这个文本（例如在讨论 Gemini CLI 的用法时），可能误判为 idle。

**建议**：pattern 已经配合 !hasSpinner 检查，误判概率低。可接受。

#### LOW: P2P completion event 泄露讨论内容

```typescript
timelineEmitter.emit(run.initiatorSession, 'assistant.text', {
  text: `**P2P Discussion Complete** (${run.id})\n\nFile: \`${run.contextFilePath}\`\n\n${run.resultSummary ?? '(no summary)'}`,
```

resultSummary（最多 2000 字符）通过 timeline 发送到所有连接的浏览器。如果讨论内容包含敏感信息，所有查看该 session 的用户都能看到。

**缓解**：这是预期行为（讨论结果应该可见）。只发 summary 而非全文已经是改进。

### 总结

没有发现 Critical 级别漏洞。主要建议：
1. temp file 创建时使用 mode: 0o600
2. timeline emitter 的 temp file 路径解析增加目录白名单验证

## 2p3z423w — Audit (hop 1/2)

### 审计结论

没有看到立即可利用的 Critical 漏洞，但当前这批改动里有几处明确的安全/逻辑风险，尤其集中在：
- temp prompt file 的读写链路
- P2P orchestrator 的“失败即跳过”策略
- timeline 中对 prompt file 的自动解引用

下面按严重度列出。

### HIGH 1: timeline 可读取任意匹配命名的本地 prompt 文件

**位置**：src/daemon/timeline-emitter.ts:11-13, 48-55

当前逻辑：
- 只要 user.message 文本匹配
  - Read and execute all instructions in @(.+\.imcodes-prompt-[0-9a-f]+\.md)
- 就会直接 readFileSync(tempMatch[1], 'utf-8')
- 并把文件内容替换进 timeline payload

风险点：
1. 这里没有做目录白名单校验
2. 正则允许任意前缀路径，只要求文件名后缀符合 .imcodes-prompt-<hex>.md
3. 如果攻击者能构造一条进入 timeline 的 user.message，就可能诱导 daemon 读取：
   - /tmp/.imcodes-prompt-xxxx.md
   - {projectDir}/.imcodes-prompt-xxxx.md
   - 甚至其他任意目录下符合命名的文件

影响：
- 本地 prompt file 内容会被注入 timeline
- 再通过 WS 广播给浏览器端观察者
- 这本质上是一个本地文件内容外泄通道

建议：
- 只允许读取受控目录：
  - /tmp/
  - projectDir 下的根级 .imcodes-prompt-*
- 更稳的做法：不要从 user text 反解文件路径，改成发送结构化字段，例如：
  - { tempFilePath, tempFileTrusted: true }
- 至少应在 timeline-emitter.ts 增加：
  - path.resolve()
  - 目录前缀白名单检查

### MEDIUM 2: temp prompt file 默认权限未显式收紧

**位置**：src/agent/tmux.ts:83-88

当前实现：
- await fsp.writeFile(filePath, keys, 'utf-8')
- 120 秒后删除

风险点：
- 没有显式设置 mode: 0o600
- 实际权限取决于进程 umask
- 对 /tmp/.imcodes-prompt-*.md 这种文件，存在同机读取窗口

影响：
- 在多用户机器上，其他本地用户可能读取 prompt 内容
- prompt 里可能包含：
  - 讨论上下文
  - 计划内容
  - 用户请求
  - 文件引用内容

建议：
- 改为：
  - writeFile(filePath, keys, { encoding: 'utf-8', mode: 0o600 })
- 如果是 projectDir 内文件，也建议同样使用 0o600

### MEDIUM 3: P2P hop 的错误处理已从“fail closed”变成“fail open”

**位置**：src/daemon/p2p-orchestrator.ts:351-360, 420-441

当前实现：
- sendKeys 失败 → retry 一次，之后 skip hop
- idle without growth → retry 一次，之后 skip hop
- timeout → skip hop
- 整个 run 不 fail，继续后续 hop / summary

风险点：
- 这会把“某一跳根本没执行”伪装成“run 仍然 completed”
- 最终 summary 可能基于不完整上下文继续产出结果
- 用户或上层系统不一定能感知某个 hop 被跳过了

影响：
- 这是逻辑完整性风险，不是权限漏洞
- 对需要严谨审查的场景，可能生成“看起来完成，但实际缺失部分参与者意见”的结果

建议：
- 至少把 skipped hop 记录进 run state / summary
- 更好：增加 partial / degraded 状态，而不是继续标成普通 completed
- 如果 continue 允许 skip，也应在最终 assistant.text 明示哪些 hop 失败或超时

### MEDIUM 4: summary 事件会把讨论摘要直接广播到 initiator timeline

**位置**：src/daemon/p2p-orchestrator.ts:320-325

当前实现：
- completed 后发一条 assistant.text
- 内容包含：
  - run id
  - context file path
  - run.resultSummary（最后 2000 字符）

风险点：
- 这段内容会进入 timeline，并广播给该 session 的观察者
- 如果讨论文件里包含敏感内容，仍会通过 summary 外泄一部分

影响：
- 风险低于全文广播，但仍是实际数据暴露面

建议：
- 默认只发：
  - run 完成状态
  - context file path / discussion id
- 由用户主动打开文件看内容
- 或只发非常短的 machine summary，不直接拼最后 2000 字符

### LOW 5: Gemini idle pattern 增加后存在误判面，但风险可接受

### LOW 6: temp prompt file 生命周期较长，且 unlink 失败静默吞掉

**位置**：src/agent/tmux.ts:88, 102-104

风险点：
- 120 秒保留时间偏长
- 删除失败完全吞掉
- 一旦清理异常，残留文件会留在 /tmp 或项目目录

影响：
- 增加本地残留敏感文件的概率
- 项目目录里残留 .imcodes-prompt-* 也可能污染工作区

建议：
- 对 projectDir 内文件缩短 TTL
- 记录 unlink failure 到 debug/warn 日志
- 在 daemon 启动时顺手清理过期 .imcodes-prompt-*

### 逻辑完整性补充观察

这意味着：

## 6i2q4d6j — Audit (hop 2/2)

### 1. HIGH: Integrity Risk - P2P Hop Skipping (Fail-Open Strategy)
The orchestrator in `src/daemon/p2p-orchestrator.ts` (lines 359, 417, 441) is designed to "skip" a hop if the agent fails to respond, times out, or fails to write to the file. 
- **Risk**: A P2P discussion relies on consecutive context building. If a "Brain" agent's planning phase is skipped but the "Worker" phase proceeds, the worker will be executing based on an incomplete or stale state. 
- **Impact**: Incorrect code changes or plans that appear to have been "discussed" but were actually silently bypassed.
- **Recommendation**: Implement a `RUN_STATUS.DEGRADED` or `FAILED` state for runs with missing hops. At minimum, the final summary must explicitly list skipped participants.

### 2. MEDIUM: Potential Information Leak - Incomplete Path Validation in `timeline-emitter.ts`
The expansion of `@.imcodes-prompt-...` instructions into file content in `src/daemon/timeline-emitter.ts` has a weak safety check for non-tmp paths.
- **Code**: `if (filePath !== resolved) return false;` (line 24)
- **Vulnerability**: This only verifies that the path is absolute and resolves to itself (no `..`). It does **not** verify that the path is within the project directory.
- **Impact**: If an attacker can create a file matching the hash pattern in any directory (e.g., `/etc/` or `.ssh/`) and trigger a timeline message pointing to it, the daemon will read and broadcast its content.
- **Recommendation**: Use `resolved.startsWith(projectDir)` for non-tmp files to ensure strict sandboxing.

### 3. MEDIUM: Privacy Risk - P2P Summary Broadcast
The emission of the last 2000 characters of the discussion to the session timeline (lines 314-325) broadcasts sensitive planning data to all observers.
- **Risk**: P2P discussions often contain internal reasoning or technical secrets.
- **Impact**: Unintentional disclosure to non-authorized observers of the session.
- **Recommendation**: Emit only the `runId` and `discussionId`. Let the user explicitly request to read the file content through a dedicated command or UI action.

### 4. LOW: Reliability - Cleanup of Temporary Prompt Files
The use of `setTimeout` for unlinking files in `src/agent/tmux.ts` (line 88) is not crash-resilient.
- **Vulnerability**: If the daemon crashes or is killed within 120 seconds of writing a prompt, the file remains indefinitely.
- **Impact**: Clutter and potential accumulation of sensitive metadata in `/tmp` or the project root.
- **Recommendation**: Implement a startup cleanup routine in the daemon to purge any orphaned `.imcodes-prompt-*` files.

### 5. LOW: Logic - Gemini Idle Detection False Positives
The pattern `/Type your message or @/m` in `src/agent/detect.ts` is relatively generic.
- **Risk**: If this string appears in the middle of a discussion or log output, the agent might be incorrectly identified as "idle" during a streaming phase.
- **Recommendation**: No immediate change needed given the `!hasSpinner` safeguard, but monitor for "premature completion" bugs in Gemini-heavy workflows.

### Summary
The audit confirms that while there are no immediate RCE vulnerabilities, the **Integrity of the P2P chain** and the **Information Disclosure via Timeline** are the most significant risks. Tightening the path validation in `timeline-emitter.ts` and refining the P2P error state are highly recommended before final release.
